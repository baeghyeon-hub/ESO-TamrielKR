# [UI/폰트] 길드 로스터 숫자 컬럼 정렬 디버깅
작성일: 2026-03-31  
상태: 해결 완료

## 문서 메타

- 문제 유형 태그: `UI 레이아웃`, `폰트 메트릭`, `Hover Inspector`, `숫자 컬럼`, `초기화 시점`
- 원인 레이어: `폰트 메트릭`, `컨트롤 구조`, `후킹 시점`
- 핵심 한 줄 요약: 길드 로스터의 챔피언 포인트 숫자 컬럼은 기본 폭 `45x20`이 너무 좁아 숫자 조합에 따라 잘렸고, `SetupRow` 후킹이 초기 생성 시점에 안정적으로 붙은 뒤 `60x20`으로 정렬이 정상화됐다.

## 1. 문제 요약

TamrielKR 적용 후 길드 로스터 오른쪽 숫자 컬럼에서 일부 값만 `20...`처럼 잘려 보이는 현상이 있었다.

대표 증상:

- 어떤 행은 `2293`이 정상 표시됨
- 어떤 행은 `2065`, `2588` 같은 값이 `20...`, `258...`처럼 잘림
- 같은 컬럼, 같은 폰트, 같은 행 높이를 쓰는데도 결과가 달라 보임

초기에는 다음 두 가능성이 모두 있었다.

1. CP 아이콘과 숫자 라벨이 너무 붙어 있어 시각적으로 잘리는 것처럼 보이는 경우
2. 숫자 라벨 박스 자체가 너무 좁아서 특정 숫자 조합만 넘치는 경우

## 2. 첫 관찰: 같은 컬럼인데 숫자마다 결과가 다름

Hover Font Inspector로 길드 로스터 행을 확인한 결과, 잘리는 행과 안 잘리는 행 모두 같은 라벨 구조를 사용하고 있었다.

대표 관찰값:

- `ZO_GuildRosterList1Row4Level`
  - font=`ZoFontGame`
  - file=`TamrielKR/fonts/univers57.slug`
  - size=`18`
  - dims=`45 x 20`
- `ZO_GuildRosterList1Row2Level`
  - font=`ZoFontGame`
  - file=`TamrielKR/fonts/univers57.slug`
  - size=`18`
  - dims=`45 x 20`

즉 문제는 "어떤 행만 다른 폰트를 쓴다"가 아니었다.  
같은 박스 `45x20`, 같은 폰트 `ZoFontGame`, 같은 크기 `18`을 쓰는데 숫자 조합에 따라 결과가 달랐다.

## 3. 결정적 단서: textDims 비교

문제를 좁히기 위해 Font Inspector에 `textDims`를 추가했다.  
이걸로 "라벨 상자 크기"와 "실제 렌더된 텍스트 폭"을 직접 비교할 수 있게 됐다.

대표 관찰값:

- `1820`
  - dims=`45 x 20`
  - textDims=`44 x 21`

이 값은 매우 중요했다.

- `1820`은 폭 `45` 안에 `44`로 간신히 들어간다.
- 그러면 `2065`, `2588`처럼 더 넓은 숫자 조합은 폭 `45`를 넘겨서 잘릴 가능성이 매우 높다.

이 시점에서 1차 결론은 다음과 같았다.

- 원인은 아이콘보다 숫자 라벨 박스 폭 부족 쪽에 더 가깝다.
- `ZoFontGame` 숫자는 고정폭이 아니라 조합별 렌더 폭 차이가 있다.
- `45x20`은 경계값에 너무 가깝다.

## 4. 왜 `SetWidth`가 처음에는 안 먹었나

초기 보정은 `GuildRoster.lua`에서 `ZO_GuildRosterList%d+Row%d+Level` 패턴의 라벨을 직접 찾아 폭을 늘리는 방식이었다.

초기 시도:

- `SetWidth(60)`
- `SetDimensions(60, 20)`
- `SetDimensionConstraints(60, 20, 60, 20)`

하지만 Inspector 출력은 계속 `dims=45 x 20`으로 남았다.

이건 두 가지를 뜻했다.

1. 라벨을 우리가 바꾸더라도 길드 로스터가 행 셋업 과정에서 다시 원래 레이아웃으로 덮어쓰고 있을 수 있다.
2. 폭은 단순 프로퍼티가 아니라 행 템플릿의 앵커/컬럼 구조에 의해 마지막에 재결정될 수 있다.

즉 "전역 스캔으로 나중에 고치기"만으로는 불안정했다.

## 5. 디버그 애드온 개선 과정

이번 사례 때문에 `TamrielKRFontInspector`를 계속 확장했다.

추가된 기능:

- 자식 텍스트 라벨 후보 자동 수집
- 직접 라벨에 마우스를 올렸을 때 부모 행의 관련 후보도 같이 표시
- `dims`와 `textDims` 동시 출력
- `anchors` 요약 출력
- 에러가 나면 빈 패널 대신 에러 메시지 본문 출력

이 과정에서 실제로 잡힌 문제도 있었다.

- `GetAnchor()`가 `relativeTo = 0` 같은 숫자를 반환하는 경우가 있었음
- 이를 컨트롤처럼 취급하면 인스펙터가 죽었음
- 그래서 비컨트롤 값은 문자열로 안전하게 처리하도록 수정함

이 디버그 애드온 덕분에 단순 추측이 아니라, "현재 컬럼 폭이 실제로 얼마인지", "글자폭이 얼마나 먹는지", "후킹이 실제로 반영됐는지"를 화면에서 바로 확인할 수 있었다.

## 6. 해결 방향 변경: 행 셋업 후킹

길드 로스터는 전역 스캔보다 "행이 그려지는 순간" 보정하는 쪽이 맞았다.

최종적으로 적용한 방향:

- `GUILD_ROSTER_KEYBOARD.SetupRow` 뒤에 `SecurePostHook`
- 각 행의 `Level` 라벨을 바로 찾아 보정
- 추가로 짧은 지연 호출을 한 번 더 걸어 마지막 레이아웃 덮어쓰기를 흡수

핵심 파일:

- `GuildRoster.lua`

이 방식으로 바꾼 뒤, 점검 후 재접속한 fresh session에서 결과가 확실히 달라졌다.

## 7. 최종 관찰: 재접속 후 정상 정렬

점검 이후 재접속한 상태에서 같은 길드 로스터를 다시 확인했을 때, 숫자 컬럼이 깨끗하게 정렬되어 있었다.

대표 관찰값:

- `ZO_GuildRosterList1Row12Level`
  - text=`2669`
  - dims=`60 x 20`
  - textDims=`41 x 20`

이 값은 해결 상태를 잘 보여준다.

- 예전: `45 x 20`
- 해결 후: `60 x 20`
- 실제 글자폭: `41 x 20`

즉 숫자 컬럼이 더 이상 경계값에 걸리지 않고 안전 여유를 가지게 됐다.

## 8. 왜 "재접속 후" 정상화됐나

이번 케이스의 중요한 포인트는 "코드가 맞아도 이미 떠 있는 리스트에는 즉시 안정적으로 안 먹을 수 있다"는 점이다.

가장 그럴듯한 해석은 다음과 같다.

1. 길드 로스터 행 컨트롤은 화면 진입 전에 이미 일부 생성/캐시될 수 있다.
2. 뒤늦게 전역 스캔으로 보정하면 다음 레이아웃 업데이트에서 다시 원래 값으로 덮일 수 있다.
3. 재접속 후에는 길드 로스터가 fresh state에서 다시 만들어진다.
4. 이때 `SetupRow` 후킹이 처음부터 붙어 있으므로 `Level` 컬럼 폭 보정이 안정적으로 적용된다.

즉 이번 이슈는 단순 폰트 문제만이 아니라 **컨트롤 초기 생성 시점**도 핵심 원인 중 하나였다.

## 9. 최종 원인 정리

최종적으로 보면 원인은 세 겹이었다.

### 원인 1. 숫자 컬럼 기본 폭이 너무 좁았다

- 기본 `Level` 라벨 폭은 `45`
- `1820`이 `textDims=44`로 겨우 들어가는 상태였다
- 그래서 숫자 조합이 조금만 넓어도 바로 잘릴 수 있었다

### 원인 2. 숫자는 조합마다 폭이 달랐다

같은 자리수라도 `1820`, `2065`, `2588`, `2293`의 실제 렌더 폭이 같지 않았다.  
그래서 "같은 행 크기인데 누구는 되고 누구는 안 되는" 현상이 생겼다.

### 원인 3. 후킹 시점이 늦으면 보정이 덮어써질 수 있었다

전역 스캔 후 보정만으로는 레이아웃 재적용을 항상 이길 수 없었다.  
행 생성 직후를 잡는 `SetupRow` 후킹이 들어가야 안정적으로 고정됐다.

## 10. 해결 상태

현재는 해결 완료로 판단한다.

근거:

- 점검 후 재접속 상태에서 숫자 컬럼이 정상 정렬됨
- Font Inspector에서 `Level` 컬럼이 `60 x 20`으로 확인됨
- `2669` 같은 4자리 값도 정상적으로 한 줄 안에 들어감

## 11. 앞으로 비슷한 문제를 만나면

같은 유형의 UI 숫자 컬럼 깨짐을 만나면 다음 순서로 가면 된다.

1. Hover Font Inspector로 문제 행을 잡는다.
2. `dims`와 `textDims`를 같이 본다.
3. 글자폭이 경계값에 걸리는지 먼저 확인한다.
4. 전역 스캔보다 `SetupRow`나 해당 UI의 실제 생성 함수 후킹을 우선 고려한다.
5. 가능하면 재접속 또는 fresh scene 진입 상태에서도 결과를 다시 확인한다.
